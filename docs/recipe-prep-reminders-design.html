Recipe Prep Reminders

Building on the Requirements & Acceptance Criteria step. After user feedback, this proposal now presents two alternative approaches:

Option A — Dedicated "Before you start" section (separate from cooking steps).
Option B — Prep items live inline inside the recipe's method list, flagged with a "Prep" badge.

A comparison & recommendation appears at the end. Both options share the same edit-form interaction patterns and accessibility properties.

The core question: are prep tasks their own activity (do them all at once before you cook) or part of the recipe flow (fit them in where they belong)?
  • Option A answers: their own activity. Prep is a checklist you run through before turning on the hob.
  • Option B answers: part of the flow. Prep is just a kind of step — sometimes it's first, sometimes it's between two cooking steps (e.g. "warm the tin" sits naturally between "make the batter" and "pour the batter").
The user's example tasks point both ways: "boil the kettle" and "tea towel on the radiator" feel pre-cook; "put lard in the tin and heat" is genuinely mid-recipe. That's why both options are worth seeing in mockup form before deciding.

Option A · Separate "Before you start" section

Option A · Design principles & decisions
Resolving the open questions from the Requirements step:
  • Snapshot vs reference (data model) — Reminders are snapshotted onto meal_items as a JSONB array, the same as ingredients. This protects the planner from broken state if a user edits or deletes the source recipe mid-cook.
  • Visual treatment in cooking mode — Reminders are a distinct, separately-labelled section above the numbered steps, not a "step 0". Reasoning: prep tasks have a different mental model (do them now, all at once, before you start) and benefit from a visually distinct container so they don't compete with the linear flow of method steps.
  • Empty state — Section is hidden entirely in read & cooking views when there are no reminders. In edit mode it shows a single ghost-button "Add a reminder" affordance.
  • Visual identity — Reminders use a soft cream-tinted card (--reminder-soft) with a small "lightbulb" mascot-friendly icon, distinct from ingredients (which use the secondary card colour) and steps (which use plain rows). This applies Gestalt similarity within the section and Gestalt proximity to keep them grouped without a heavy outline.
  • Once everything is ticked — The card collapses to a slim "All prep done ✓" strip and the cooking view auto-scrolls to step 1, applying Hick's Law (reduce visible options once they're complete).
Option A · 1. Recipe edit form
Reminders sit between Ingredients and Instructions in the edit form (matches the order they'll be read in cooking mode). Each row has a drag handle (Fitts: 44px tall, easy thumb target on mobile), an inline text input, and a delete affordance. A persistent ghost button at the bottom lets the user add another. Character counter appears only when ≥ 100/140 chars used.

Option A · 4. Interaction flow & states
State map (per Compass UX heuristic — handle every state explicitly):
  • Empty (no reminders) — Edit form: educational empty card + one ghost button. Read view: section hidden. Cooking mode: section hidden.
  • Populated, none ticked — Edit form: list of editable rows. Read view: list of read-only rows. Cooking mode: full reminder card at top with "0 / N" progress.
  • Partially ticked — Cooking mode shows ticked items at reduced opacity with strikethrough; progress label updates ("2 / 3").
  • All ticked — Card transitions to compact "Prep done · N of N" bar with a tiny chevron to re-expand if needed. Progress badge turns green.
  • Editing while in cooking mode (out of v1 scope) — Cooking-mode reminders are read/tick only; if the user wants to add one mid-cook they exit to edit. (Documented as a v2 nice-to-have if needed.)
  • Loading — Skeleton: a 32px tall placeholder bar inside the reminder section while the recipe is fetched. Re-uses the existing skeleton pattern from ingredients.
  • Error (save failed) — Inline toast "Couldn't save your reminder, try again" — same as the existing recipe-save error pattern.
Accessibility: Each tickable row is a <button> with aria-pressed, not a styled div. Reminder card has role="region" and aria-label="Prep reminders". Drag handle has aria-label="Drag to reorder"; keyboard reorder via Up/Down with focus on the handle (matches the React DnD pattern already in the project). Contrast checked: reminder text on --reminder-soft is 9.2:1 (AAA). Mobile constraints: 360px width verified — text wraps to 2 lines comfortably; drag handle and delete buttons are 44px tap targets; no horizontal overflow.
Option A · 5. Data model implications
  • Add reminders JSONB column to recipes — array of { id: string, text: string }. id is a stable key for reorder/animation; default [].
  • Add the same reminders column to meal_items (snapshot at add-to-plan time, mirroring how ingredients is handled).
  • Type validation in /types/database.ts and serialiser/deserialiser helpers in /lib.
  • Cooking-mode tick state lives in the existing itemCheckState Map structure — extend the per-item shape to include checkedReminders: Set<number> alongside the existing checkedIngredients and completedSteps. Persistence is automatic via the sessionStorage we just shipped.

Option B · Prep tasks inline within method

Concept. No separate section. Instead, each item in the recipe's method list has a type: either a normal cooking step or a prep task. Both live in one numbered list. Prep tasks are visually flagged with a small green "PREP" badge and a left-edge accent — similar to how Linear flags issues by type, or how Notion uses inline callouts. The user places them where they belong in the flow — sometimes that's at the very top (mise en place), sometimes between steps (warm the tin between mixing and pouring).

Why this works. It treats prep as part of the recipe rather than as a separate concept the user has to manage. From the cook's point of view, you read the recipe top to bottom and do whatever each item asks — no switching between two lists. The "PREP" badge gives you the same scannability as a separate section (you can pick out "what do I need to do before I start" by skimming for green badges) without splitting the data model in two.
Option B · 1. Recipe edit form
Method becomes a single list. Each row has a two-state toggle on the right (STEP / PREP) — segmented-control pattern from iOS settings, familiar from Jakob's Law. Default new rows are STEP; the user taps once to flip to PREP. Drag handle reorders within the same list, so prep can sit anywhere.

Option B · 3. Cooking mode
Cooking mode renders the same single list, every item tickable. Prep items keep their badge + accent so the user can mentally batch them ("let me knock out the green ones first"), but they appear at their natural position in the recipe. The first unchecked item is highlighted as the active step, regardless of type — this preserves the existing cooking-mode interaction and keeps the user moving forward through the recipe.

Option B · 4. Data model implications
Smaller and simpler than Option A. Today recipes.instructions is a single text/markdown field. Option B requires either:
  • Approach 1 (lighter, recommended): Keep instructions but restructure to a JSONB array of { id: string, text: string, type: 'step' | 'prep' }. A migration converts existing string-based instructions into type: 'step' entries (split on newlines or list markers — fallback safe).
  • Approach 2 (least invasive): Keep instructions as text/markdown and use an inline marker syntax like [prep] Boil the kettle that the renderer parses. No schema change, but parsing logic + lossy round-trip if the user edits raw markdown.
No changes to meal_items beyond what already exists (instructions are already snapshotted there). Tick state in cooking mode reuses the existing completedSteps: Set<number> on itemCheckState with no extension — every list item is a "step" from the tick-state's point of view.
Comparison & recommendation

Option A · Separate "Before you start" section

  • Strong "mise en place" affordance — see all prep at a glance before starting.
  • Visually batched: the user can tick all prep before turning on the hob.
  • Existing instructions field stays untouched.
  • Two lists to manage in cooking mode — small context-switch cost.
  • Mid-recipe prep ("warm the tin between mixing and pouring") doesn't fit naturally — has to live above all the cooking, away from where it's actually used.
  • New JSONB column on recipes + meal_items — bigger data-model footprint.
  • Maximum of 20 items per recipe — extra constraint to enforce.

Option B · Prep inline within method

  • Single source of truth for "what to do" — read top-to-bottom, no switching.
  • Prep can sit at the right position in the flow (e.g. "warm the tin" between mixing and pouring) — matches how cooks actually think.
  • Smaller new UI surface — extends existing steps rather than adding a new section.
  • Cooking-mode tick state reuses the existing completedSteps Set with no extension.
  • Migration path for existing recipes is automatic — every existing instruction becomes a type: 'step' entry.
  • No batched "see all prep at once" view (partly mitigated by green badges being scannable).
  • Requires restructuring recipes.instructions from text → JSONB array (or accepting lossy markdown markers).
Compass's recommendation: Option B.

Three reasons:
  1. It matches how cooks think. The user's three example tasks — boil kettle (pre-cook), warm tin (mid-recipe), tea towel (pre-cook) — already span the whole timeline of a recipe. Option A forces them all to live above the cooking; Option B lets each one sit where it belongs.
  2. It's less code, less data, less UI surface. Reuses the existing tick-state model, the existing snapshot to meal_items, the existing cooking-mode rendering loop. Option A introduces a parallel set of all of those.
  3. The "scan for prep" use case is preserved by the badge. A green PREP chip on a few rows is easily scannable on a 360px screen — you don't lose the "what do I need to do before I start" affordance, you just don't need a separate section to get it.
Caveat / when to prefer Option A: if you imagine future versions where prep gets its own behaviours that don't apply to cooking steps — e.g. "remind me 10 minutes before serve time", or "Alexa, read me my prep list before I start" — then Option A's separation pays off because prep has its own data home to hang those features off. If those are likely v2 features, Option A is the more defensible long-term shape even though it's heavier today.
