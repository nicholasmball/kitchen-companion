Chef chat — “Add to plan” button

Building on the Requirements & Acceptance Criteria step's decisions: just-meal-item (no auto-recipe), one-tap when active plan exists, picker otherwise, sibling to Save to Recipes. This doc shows the three button states + the picker reuse + toast/error/loading states. Compact because the surface area is one component file.

State A · Active plan exists (one-tap)
Most common case. Button label is Add to "{plan name}", truncated with ellipsis if > 22 chars (after the "Add to " prefix). Tooltip on hover/long-press shows the full plan name.

Both buttons use the same primary orange variant (matches the existing Save to Recipes treatment) — they're both equally valid actions; this isn't a scenario that justifies hierarchy. Save to Recipes stays first because the natural reading order is "save permanently first, then drop into tonight's plan."

State map (Compass UX heuristic — handle every state):
  • Loading — clicking the button shows inline "Adding…" text and disables both action buttons until the mutation resolves. No spinner; the colour-faded disabled state + text change is enough for < 1 s mutations.
  • Optimistic — could update immediately, but planner data isn't on screen so there's no UI to reconcile; await the round-trip + use the toast as confirmation (familiar pattern from the sync push toast).
  • Error — destructive toast: "Couldn't add to plan. Try again?" with a retry-on-tap action. Button returns to its default state.
  • Empty (no recipe in message) — both buttons are hidden (existing behaviour for Save to Recipes via containsRecipe).
  • Picker open during streaming — if the AI is still streaming when the cook taps, the button works on whatever recipe content has arrived so far (rare; the cook would normally wait until the message is complete because the button text only renders post-streaming).
  • Concurrent — if the cook taps both Save to Recipes AND Add to plan rapidly, both fire independently and both succeed (no shared state).
Accessibility:
  • aria-label on the Add to plan button uses the full plan name (no truncation): "Add “Sunday Roast — that 4-hour beef one” to plan".
  • Both buttons keyboard-reachable; focus order: Save to Recipes → Add to plan.
  • Picker: existing dialog already follows shadcn focus-trap conventions.
  • Touch targets: ≥ 36px height + 0.5rem horizontal padding gives ≥ 44px effective hit area on iOS/Android (matches existing button sizing).
  • Stacked layout on narrow bubbles avoids horizontal overflow at 320 – 380 px wrapper widths.
Implementation notes for Atlas:
  • chat-message.tsx: add a sibling button under {showSaveButton && !saved && ...}. Wrap both in flex flex-col sm:flex-row gap-2 so they stack on narrow bubbles and sit side-by-side once there's room.
  • Determine button mode by reading useMealPlans: activePlan (one-tap) vs mealPlans.length (picker vs picker-in-create-mode).
  • One-tap path calls addMealItem(activePlan.id, payload) directly — no dialog open.
  • Picker path opens the existing AddToPlanDialog with the parsed recipe data; extend its RecipeData prop to optionally accept ingredients and notes so they round-trip into the meal item (current dialog drops them).
  • AddToPlanDialog needs a tiny tweak: when called with no plans, default mode to 'create' and pre-fill newPlanName from the recipe title.
  • Toast uses the existing sonnerToast, mirroring the Phase 2 push toast — same UX vocabulary across the app.
What this design deliberately does NOT add:
  • No "Save to recipes & planner" combined option in v1 — the cook can hit both buttons.
  • No mid-flow editing of the meal item — they edit after via the planner.
  • No serve-time selector when adding to an existing plan — uses the plan's existing serve time.
