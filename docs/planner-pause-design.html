Planner pause / push-back / padding

Building on the Requirements & Acceptance Criteria step's locked v1 set (A · pause + B · push-back + D · planning padding). The visual "PAUSED" state has to be impossible to miss for a cook glancing back at a phone on a flour-covered counter — that's the centrepiece of this design. Mockups assume a real-cook context (messy hands, time pressure, peripheral vision).

1 · Running plan — Pause + Push-back affordances
Pause and Push-back live in a dedicated bar directly under the page header — not buried in a menu, because (a) they're high-stakes mid-cook actions and (b) thumb reach matters. Both buttons are 56 px tall (raised from project standard 44 px per requirements — messy hands deserve more target).
  • Pause uses the amber "warning" colour palette so its triggered state (the banner) is colour-consistent.
  • Push back serve time is a quieter outline button — used less often.
  • Both visible only when plan.serve_time is set (i.e. the plan has actually been scheduled — no point pausing an unscheduled plan).

2 · Plan paused — the centrepiece banner
When paused, the bar morphs into a full-bleed amber banner with: pause icon (high contrast white-on-amber), elapsed timer (font-variant-numeric: tabular-nums, ticks every second), large prominent Resume button, and a small reminder about the heat. Subtle radial pulse animation (3 s loop) makes it visually alive without being distracting — Compass-tested heuristic for "the system is waiting on you" states.

Why amber, not red? Red signals "something is wrong" — it'd raise stress. Amber says "we've stopped intentionally" — calm and recoverable. The banner shouts visually without shouting emotionally.

Tab title gets prefixed with ⏸ Paused — so users glancing at a browser tab on a laptop also see it (e.g. cook ducks back to the office mid-cook).

3 · Push back serve time menu
Tapping Push back opens a popover anchored to the button. Five tap targets (4 quick + 1 custom), each ≥ 48 px tall. Each option shows the resulting serve time so the cook can pick by destination, not just by delta — recognition over recall.

Quick picks one-tap apply. "Pick a new time…" opens a small datetime input dialog. Push-back is also accessible inside the paused banner (subtle "or push the serve time" link below the Resume button) for the case where a quick pause turns into a longer delay.

4 · Plan create / edit form — buffer minutes field
The padding field lives in the plan create / edit form, near the serve-time field (where the timing concept is already in the cook's head). Surrounded with a soft green dashed outline so it visually says "this is the contingency cushion" without needing extra labels (Gestalt similarity to the "before you start" / action-item theme established in the recipes feature). Default is 0; help text recommends 10–15.

Showing the resulting "start cooking 10 minutes earlier than the plan strictly requires" microcopy (computed live from the field value) helps cooks understand what they're configuring.

5 · State map (handle every state)
  • Plan running, no serve_time set — Pause / Push-back hidden (nothing to pause / push). Help text in the timeline area: "Set a serve time to enable pause and push-back."
  • Plan running, serve_time in the past — Pause / Push-back still visible (cook may run a plan after its scheduled time). Push-back works normally.
  • Plan paused — banner visible. Pause button replaced by the banner. Push-back still tappable inside the banner.
  • Resume tap during paused — banner fades out (200 ms), timeline times update, success toast: "Resumed. Serve at 6:08 PM".
  • Network failure during pause/resume — optimistic UI updates immediately; if the mutation fails we roll back + toast: "Couldn't pause. Try again?". The cook isn't left in an unknown state.
  • Multi-tab — if the user has the planner open in two tabs, real-time subscription (already in the project) syncs the pause state across tabs.
  • Plan deleted / made inactive while paused — pause state is lost (acceptable; the cook explicitly removed the plan).
  • Cooking-mode entered from a paused plan — banner persists; step checkboxes still tickable (paused affects timing, not interactions).
  • Cook taps Pause repeatedly — debounced; only the first tap fires the mutation.
  • Push-back to a time before now — disallowed; quick-pick still works (since adds positive delta), custom datetime input has a min attribute pinned to now.
Accessibility:
  • Pause / Resume / Push-back are <button> elements with descriptive aria-labels ("Pause the meal plan", "Resume the meal plan, currently paused for 4 minutes 23 seconds", "Push back the serve time").
  • The paused banner has role="status" + aria-live="polite" so screen readers announce the pause state.
  • The elapsed timer updates via aria-live="off" to avoid flooding the screen reader with second-by-second announcements; it's decorative for sighted users.
  • Amber on amber-banner contrast: white-on-#C99846 = 4.6:1 (AA). Tested.
  • Pulse animation respects prefers-reduced-motion — disabled when the user has reduced-motion preference set.
  • Keyboard: Pause / Resume reachable via Tab; Esc closes the push-back popover.
  • Tap targets ≥ 56 px tall (raised from project standard 44 px per real-cook requirements).
6 · Data model + implementation notes for Atlas
Schema (3 new columns on meal_plans):
  • paused_at: timestamptz | null
  • total_pause_seconds: int default 0
  • padding_minutes: int default 0 (with check: padding_minutes >= 0 AND padding_minutes <= 120)
Timing-calculator changes (lib/timing-calculator.ts):
  • Effective offset = total_pause_seconds + (now - paused_at if paused else 0). Add this to every event time.
  • padding_minutes shifts the first event earlier by N minutes — i.e. the calculator's "earliest start" point becomes serve_time - total_required_time - padding_minutes.
  • Both adjustments compose cleanly: effective_event_time = serve_time - (countdown_to_event) - padding_minutes + pause_offset.
Notifications:
  • On Pause: cancel all scheduled notifications for this plan via the existing helper in lib/notifications.ts.
  • On Resume: re-schedule using the recomputed event times.
  • If the cook resumes mid-event (e.g. Resume happens 3 seconds after the cook_start should have fired), do NOT fire a missed notification — just continue from "now" forward.
UI extraction:
  • New <PausedBanner> component — encapsulates the banner + elapsed ticker + Resume button + heat hint. Used by both the planner page and cooking-mode views.
  • New <PushBackMenu> component — popover with quick-pick + custom-time options. Reused inside the banner.
  • Pause + Push-back buttons live in a new <PlanControlsBar> component below the page header.
