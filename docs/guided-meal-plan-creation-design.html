Guided Meal Plan Creation

UX Design Document — Building on Requirements & Acceptance Criteria (Step 1)

Design Rationale
Problem: Creating a meal plan currently involves a dialog for basics, landing on an empty page, then figuring out separate buttons to add items. New users don't understand what a meal plan does or how timing works.

Solution: A full-page 3-step wizard (R1) at /planner/new that guides users from naming to adding dishes to reviewing — all before creating the plan. This follows Hick's Law (fewer choices per step) and progressive disclosure (show what's relevant now).

Key decisions:
• Full-page wizard, not dialog: A wizard gives enough space for recipe cards, search, and the review timeline. Dialogs are too cramped for this.
• Step indicator: Numbered dots (1-2-3) with labels. Completed steps get a green check. Users always know where they are.
• Recipe quick-add cards: Tap a recipe card to add it — card gets a green checkmark. No intermediate confirmation. This is the "one tap to add" principle from the design guide.
• Custom item mini-form: Only 3 fields (name, cook time, method) vs the full 9-field form. Reduces friction for simple items like "Frozen peas — 5 min — hob".
• Mini timeline in Review: Shows a condensed preview of when things happen. Builds confidence that the plan makes sense before committing.
• Batch create: Nothing is persisted until "Create Plan" in Step 3. User can exit at any time without side effects.

Component Architecture
New route: app/(main)/planner/new/page.tsx — hosts the wizard
New component: components/planner/create-plan-wizard.tsx — the full wizard with internal step state

Internal state:
step: 1 | 2 | 3 — current wizard step
planName: string — from Step 1
serveDate: string — from Step 1
serveTime: string — from Step 1
selectedItems: WizardItem[] — items added in Step 2 (recipe-based or custom)

WizardItem type:
{ name, cook_time_minutes, prep_time_minutes, rest_time_minutes, cooking_method, temperature, temperature_unit, instructions, recipe_id, ingredients }

On "Create Plan":
1. Call createMealPlan({ name, serve_time, is_active: true })
2. For each item, call addMealItem(planId, item)
3. Navigate to /planner/{planId}

Planner list page change:
Change "Create Meal Plan" button from opening dialog → navigating to /planner/new
Accessibility
• Step indicator: aria-current="step" on active step, aria-label="Step 1: Name and Time"
• Recipe cards: role="button", aria-pressed={isAdded}, aria-label="Add Roast Chicken to plan"
• Form inputs: visible labels (not just placeholders), proper htmlFor linking
• Navigation: Back/Next buttons always visible, keyboard tab order follows visual order
• Close button: aria-label="Cancel and return to planner"
• "Next" disabled state: includes text explaining why ("Add at least one dish to continue")
• Mini timeline: uses list semantics with role="list"
