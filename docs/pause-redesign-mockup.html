Mockups for the two pause bugs being fixed together. Red times show the broken behaviour today; stable times show the post-fix behaviour. Both fixes ship in one PR because they touch the same code.

Bug 2 · Times jump forward on refresh during pause
During pause, the displayed timeline includes the current elapsed pause window in its offset. That offset is recomputed on every render. So a hard refresh (or any state change that re-renders the page) picks up a larger pause window and shifts every event forward.

Fix: drop the current pause window from the display calculation. Times stay frozen at whatever they were when pause was tapped. Only the elapsed counter at the top of the banner ticks.

Bug 1 · Resume should not auto-shift
Today, Resume always pushes every event forward by exactly the pause duration — even if the cook had slack ahead and didn't actually consume any of it. That conflates pause with push-back.

Fix: Resume just unfreezes. Schedule unchanged. If the cook fell behind (a scheduled event passed during pause), a toast offers a one-tap push-back.

Two scenarios shown below: (1) cook was within their slack window — schedule untouched; (2) cook fell behind — toast offers catch-up push-back.

Resume toast lifecycle
  • Trigger: cook taps Resume AND ≥ 1 scheduled event has passed during pause.
  • Lifetime: ~12 seconds (Sonner default for toasts with action). Auto-dismiss after that — gone.
  • Manual dismiss: ✕ button on the toast. Same as auto-dismiss.
  • Manual accept: tap "Push back" → invokes existing push-back flow with the appropriate delta. Toast closes.
  • If ignored: gone in ~12 s. The cook still has clear cues — past-due events stay flagged in the timeline + the Push-back button in the controls bar is always there for manual catch-up.
  • Re-trigger: only fires once per resume action. If the cook pauses again and resumes again, a new evaluation runs.
Net effect: the toast is a one-time helpful nudge, not a sticky alert. Cook is never trapped staring at a banner they don't want.
All states (post-fix)
  • Pause tapped: banner appears, notifications cancelled, displayed timeline frozen at the moment of pause.
  • During pause, page re-renders for any reason: timeline times unchanged. Only "Paused for X:XX" counter advances.
  • Resume tapped, no scheduled events passed: banner disappears, schedule untouched, notifications rescheduled at original times. No toast.
  • Resume tapped, ≥1 scheduled events passed during pause: banner disappears, schedule still untouched, toast appears: "You're {N} min behind. Push serve to {newTime}?". One-tap accept invokes the existing push-back flow (which anchors in-flight items correctly via the cook_end_override fix). Dismiss leaves the cook to handle manually.
  • Multiple pause/resume cycles: each cycle is independent. Schedule only changes when the cook explicitly hits Push back.
  • Pause + push-back combined: push-back works as today (already correctly anchors in-flight items). Pause has no effect on the push-back delta.
Implementation summary (for Atlas)
Bug 2 fix (refresh shift):
  • getPauseOffsetSeconds(plan) drops the now - paused_at term — returns just total_pause_seconds.
  • (After Bug 1 fix, total_pause_seconds is always 0, so this becomes a no-op for pause; the function survives for the padding side or could be removed.)
  • Countdowns ("in 3m") need an effective now. When the plan is paused, pass paused_at as the effective now to formatTimeUntil so countdowns also freeze.
Bug 1 fix (no auto-shift):
  • handleResume sends { paused_at: null } only — does NOT mutate total_pause_seconds.
  • After successful resume, walk the timeline (computed from unchanged serve_time) and check whether any non-completed event's time is now in the past. If so, surface the "you're behind" toast with the appropriate delta.
  • Toast action button calls the existing handlePushBack({ addMinutes: deltaMin }) — which already correctly anchors in-flight items.
Schema: no changes. total_pause_seconds stays in the schema (vestigial after this change; cleanup can wait).
