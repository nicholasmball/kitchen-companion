Recipe ↔ Meal-plan sync

Today, adding a recipe to a meal plan creates a frozen snapshot. The recipe and the plan item drift independently after that point. This document explores how to give cooks visibility into that drift, plus controlled tools to pull updates from the recipe and push improvements back to it.

The cook's mental model
From watching how home cooks behave (rather than how recipe apps assume they do):
  1. Recipes are living documents. Cooks iterate every time they make something. "I've made this 20 times — now I know it needs more salt at the end." They expect their recipe to evolve.
  2. A meal plan is a single occasion. Tonight's roast, for these guests, with this oven, at this serve time. Per-occasion tweaks should NOT pollute the canonical recipe.
  3. In-flight cooks are fragile. Once you're in the kitchen with the timer running, the screen must NOT change underneath you. Anything that mutates mid-cook will be filed under "this app betrayed me."
  4. Cooks re-cook things on weekly cadence. If I improved the recipe last Sunday, I expect this Sunday's plan to use the improved version.
  5. Cooks make mid-cook discoveries. "I should have boiled the kettle first" — they want to capture the lesson back to the canonical recipe so they don't forget next time.
Design constraints that follow:
  • Storage stays as snapshot — protects the in-flight cook.
  • Sync is always manual, never automatic — the cook chooses when to act.
  • Sync is bi-directional, but each direction has different defaults and friction (see below).
  • Drift between snapshot and canonical recipe must be visible wherever the user could care.

↓ Direction 1 · Recipe → Plan

"Pull updates from the recipe into my plan"
1.1 · Stale badge on the planner
When viewing a meal plan, every meal item that was sourced from a recipe carries a hidden stamp: recipe_snapshot_at. If the source recipe's updated_at is newer than the snapshot, the meal item card shows a small "Recipe updated" badge — amber, low-noise, tappable.
  • If the source recipe hasn't changed since snapshot: no badge, zero new visual noise.
  • If the source recipe has been deleted: no badge (snapshot is the source of truth — same as today).
  • Tapping the badge opens the diff dialog (1.2).

1.2 · "Refresh from recipe" diff dialog
Tapping the stale badge opens a focused dialog showing exactly what would change. Cooks scan, they don't read — the diff uses Gestalt similarity (green = added, red strike = removed, amber = changed) so the gist is visible at a glance. Per-occasion tweaks (anything the cook changed locally on the meal item since the snapshot) are flagged as "your changes" inline, so the cook sees what they'd lose.

↑ Direction 2 · Plan → Recipe

"Save my mid-cook discoveries back to the recipe"
2.1 · When does this need to happen?
Three real-cook scenarios:
  • "I learned this while cooking." Mid-prep, you realize the recipe should say "marinate the chicken for 30 min first." You add it as an action in the meal item. Want it on the canonical recipe so next time you don't forget.
  • "Tonight-only adaptation." Skip the chili because Mum's coming. Don't want this on the canonical recipe.
  • "Improved on the fly." While reading the steps, you realize step 4 and 5 should swap. You re-order them in the meal item. Want this in the recipe.
Two of three default to "yes, save to recipe." But the third is non-trivial — and the cost of a wrong "yes" (polluting the canonical recipe) is higher than the cost of a wrong "no" (one extra edit later). Default = OFF; cook opts in.
2.2 · Save toast with "Also save to recipe?" affordance
After the cook saves any change to a meal item that originated from a recipe, show a non-blocking toast offering to push the change to the recipe. Auto-dismisses after ~6 s. Tapping "Save to recipe" opens a small confirm dialog (2.3) showing what would propagate. This is Hick's-Law friendly: one CTA, one ignore-path.

2.3 · "Save to recipe" confirm dialog
Tapping "Save to recipe" opens a confirm dialog with the same diff treatment as the upstream direction, but with the colour scheme inverted (changes are going to the recipe). The cook can deselect individual changes — useful when, e.g., they want to push the new action item but keep tonight's reduced cook time as a per-occasion tweak.

2.4 · Conflict path · recipe edited since snapshot
Edge case: cook edited the meal item AND the recipe was edited on another device since the snapshot. Now any push has to merge with already-newer recipe content. Real cooks won't deal with a 3-way merge. Reduce to two clear options:
  • "Refresh first, then push" — pulls the recipe's new state into the meal item (with diff), THEN cook can re-attempt the push. Two-step but each step is simple.
  • "Replace recipe with my version" — destructive, surfaced behind a secondary CTA with a clear warning. For the case where the cook explicitly wants their version to win.

Data-model implications
Minimal schema change — one new column on meal_items:
  • recipe_snapshot_at: timestamptz | null — set whenever the meal item is created or refreshed from a recipe. null for custom items.
Stale detection is a single comparison: meal_items.recipe_snapshot_at < recipes.updated_at. The badge is a derived UI signal, not stored.

Diff computation runs on the client at dialog-open time — compares the snapshot fields (name, ingredients, instructions, prep/cook/rest, temperature) against current recipe fields. Existing serialisers (parseInstructionItems, etc.) handle the structured comparison for steps + actions.

Push (Plan → Recipe) is a regular recipe update mutation, scoped to the fields the cook ticked in the dialog. No new endpoint needed.
All states
  • Empty / no recipe — meal item has recipe_id = null. No badge, no toast, no edit-dialog hint. Behaves exactly as today.
  • Source recipe deleted — meal item retains its snapshot, no badge, edit dialog says "Source recipe deleted" instead of the snapshot date.
  • In sync — no badge; edit dialog shows the orientation line (1.3 left).
  • Stale — amber badge; edit dialog shows the refresh CTA (1.3 right).
  • Cook just edited the meal item — toast (2.2). If they tap "Save to recipe" → diff dialog (2.3).
  • Push conflict — recipe was also edited externally → conflict dialog (2.4).
  • Loading / network failure — sync actions are optimistic with rollback. If the push fails, restore the meal item's previous state and surface a toast.
Appendix · Phase 1 readiness review
Building on the Requirements & Acceptance Criteria step's Phase 1 scope, this appendix confirms the existing design (sections 1.1–1.3 above) covers the full Phase 1 ask, and adds the loading/error/accessibility detail that the requirements step deliberately deferred to design.

Phase 1 ↔ design coverage check:
  • ✅ Stale badge on planner card — section 1.1, mockup shows in-sync vs stale states + custom-item case.
  • ✅ Diff dialog — section 1.2, covers add / remove / modify and inline local-tweak conflict.
  • ✅ Snapshot context line in edit dialog — section 1.3, both in-sync and stale states.
  • ✅ Refresh action wires through both surfaces — badge tap and edit-dialog CTA both open the same diff dialog.
  • ✅ Manual-only sync — every interaction is user-initiated.
  • ⚠️ Loading + error states — not previously specified; addressed below.
  • ⚠️ Accessibility specifics — not previously specified; addressed below.
Loading state. The diff is computed client-side from data already in memory (the planner page already has the meal item; the recipe is fetched once on dialog open). So the dialog has two visible states:
  • Recipe fetch in progress (rare — only on dialog open and only if recipe not in cache): show a 3-line skeleton inside the diff list, matching the height of an actual diff row. No spinner.
  • Refresh mutation in progress: the "Refresh" button shows inline loading state ("Refreshing…", disabled, spinner glyph). Dialog stays open until success, then closes with a toast: "Updated from recipe."
Error states.
  • Recipe fetch fails: dialog shows an inline message — "Couldn't load the latest recipe. Try again?" with a Retry button. Cancel still available.
  • Refresh mutation fails: dialog stays open, banner appears at top — "Couldn't save your refresh. Your meal item hasn't changed. Try again?" Same Retry button.
  • Recipe deleted between badge appearance and dialog open (race condition): dialog shows — "The source recipe was deleted. Your meal item is unchanged." Single "OK" button to close.
Accessibility.
  • Stale badge is a <button> with aria-label="Recipe for [item name] has been updated. Tap to review changes." — descriptive label, not just "Recipe updated" (which would lose context for screen readers tabbing through cards).
  • Badge ≥ 44×44px on mobile (Fitts's Law for thumb-targets). Tappable area extends slightly beyond the visible badge to meet the size requirement without bloating visual weight.
  • Amber colour (#C99846) on cream (#FAF6F1) measures 4.6:1 contrast — meets WCAG AA for normal text. Don't rely on colour alone — the dot + text + border-tint combine for redundancy (per Compass constraint).
  • Diff dialog uses + / − / ~ markers as well as colour, so the diff type is conveyed without relying on red/green/amber alone.
  • Dialog is keyboard-navigable: Tab cycles through diff items (informational only, not focusable interactively in Phase 1) → Cancel → Refresh. Esc closes. Focus returns to the badge that opened the dialog.
  • Local-tweak conflict banner uses role="alert" so screen readers announce it when the dialog opens.
Mobile responsive notes.
  • At 360px width, the meal-item card's right-aligned badge can wrap to a second row beneath the item name — verified in the mockup, no overflow.
  • Diff dialog uses the same bottom-sheet pattern as the existing planner recipe sheet on mobile (height: 75vh, scrollable diff list inside).
  • The "Cancel" / "Refresh" button row stays stuck to the bottom of the dialog so it's always reachable without scrolling.
Edge cases worth raising before build:
  • Recipe edited multiple times since snapshot — the diff still works because we always compare snapshot vs current. No multi-version history needed.
  • Cook opens diff dialog, recipe is edited on another device while dialog is open — accept the small inconsistency; the refresh button operates on whatever the dialog last fetched. Worst case: the cook re-opens the badge, sees a fresh diff. Not worth implementing live updates inside the dialog for Phase 1.
  • Time-zone display — snapshot date in the orientation line should respect the user's locale. Use the existing date formatter in lib/utils.ts if there is one; otherwise default to toLocaleDateString('en-GB', { day: 'numeric', month: 'short' }) matching the rest of the app.
  • "Snapshot DATE" with same-day refresh — show "today" if same calendar day, "yesterday" if -1, otherwise the date. Tiny detail but matches Linear/GitHub conventions cooks already know.
What's NOT being added here — staying in Phase 1 scope:
  • No three-way merge UI. If the recipe and meal item have both diverged, the user's tweaks are flagged but the user still picks "Refresh (overwrite)" or "Cancel" — no per-field merge in Phase 1.
  • No undo of refresh. Recipe-state-before-refresh is not stored. (If the cook hits Refresh by accident, they'd need to manually re-apply their tweaks via the edit dialog.) Acceptable given Cancel is right there and explicitly the default-positioned action.
  • No bulk refresh, no recipe-list staleness counter, no settings toggle — all deferred per Phase 1 scope.
Ready for build. No design gaps that block Atlas. The existing mockups + this appendix together cover all states required by the Phase 1 acceptance criteria.

  • Toast after save (2.2).
  • Push diff dialog with per-change opt-in (2.3).
  • Conflict dialog (2.4).
Phase 3 (polish):
  • Indicator on the recipes list showing "N plan items would update from this recipe."
  • Setting toggle: "Always ask to save changes back to the recipe" (default ON).
